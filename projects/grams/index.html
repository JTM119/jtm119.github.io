System Design
Cognitive Architecture Components

GRAMS is composed of five integrated components operating within a unified ML framework and a Redis-backed control script. Each component shares a common autoencoder encoder base, enabling simultaneous analysis across all subsystems the moment a failure is detected.

RIA
Risk Identification Algorithm
Monitors sensor streams using autoencoder reconstruction error to detect anomalies. Computes per-sensor saliency maps (0–1 normalized gradients) for crew-facing explainability.
AG
Alert Generator
Routes RIA alerts to the appropriate recipient based on urgency: crew, autopilot, maintenance log, or ground station. Outputs failure severity classification.
FS
Failure Simulator
Classifies known failure modes (KFC) and flags unknown ones (UFC/IUC). The Unknown Failure Generator (UFG) synthesizes novel failure trajectories from a single example for retraining.
DT
Digital Twin
Simulates spacecraft propulsion in 6 DOF at 1000 Hz using a custom SciPy/Fortran physics engine. Based on a 2× scaled Orion service module. Implements five failure modes drawn from historical spaceflight incidents.
AR
Action Recommender
Plans multi-step recovery sequences using Monte Carlo Tree Search (MCTS) with a Transformer-based reward model, replacing the Phase I greedy planner to enable long-horizon planning. Evaluated 100,000 candidate action sequences per run, producing identical optimal sequences across repeated runs.
Machine Learning
Unified Autoencoder Architecture

Phase II replaced per-component independent models with a unified architecture: a single autoencoder base model is trained on nominal sensor data, then component-specific head models are fine-tuned with the encoder frozen.

Unknown Failure Generation: TLRE

A bidirectional GRU encodes failed and nominal trajectories, extracts a residual, and applies it within a time window to new nominal trajectories. Operates on first and second derivatives of sensor values. Deterministic; best for sharp, localized failures.

Unknown Failure Generation: FaO-ANP

Failure-as-Operator Attentive Neural Process. Encodes a single failure example as a context vector via kernel-based attention and outputs a residual distribution (mean and variance) at each timestep. Handles gradual onset and uncertain failure timing; provides uncertainty estimates for downstream risk analysis.

Flight Testing
Orbital Validation

GRAMS was tested in two distinct orbital environments, validating both its containerized deployment strategy and live ML inference on spacecraft telemetry.

ISS / HPE Spaceborne Computer 2
DCU-1  ·  NASA PEP Network  ·  Singularity Container
Test ML models (Diffusion and Graph Neural Network) ran on SBC-2 aboard the ISS for more than two weeks continuously. Validated the containerized deployment pipeline including low-bandwidth update procedure (<12 KB patches). NASA's Space Radiation Analysis Group provided radiation dose data from the Columbus module for correlation analysis with model performance. The full GRAMS CA could not run on SBC-2 due to hardware availability constraints, but deployment and long-duration runtime were fully confirmed.
> 2 WEEKS CONTINUOUS RUNTIME
ABOVE Orbital Platform
Full GRAMS CA  ·  Live Telemetry  ·  30-Second Sensor Cycle
Full GRAMS cognitive architecture deployed and run for ~30 hours on orbit. Successfully ingested live spacecraft telemetry (CPU temperature, solar panel voltage, battery voltage, load current, latitude, longitude, altitude, orbital velocity) delivered as JSON files every 30 seconds. The RIA detected anomalies in the live sensor stream, confirming end-to-end operation of the ML pipeline in an orbital environment. The Digital Twin used a data-matching approach to predict nominal LLA trajectories without requiring retraining.

Human Factors
User Interface and HERA Testing

The UI was built as a standalone Electron application communicating with GRAMS exclusively via its existing Redis database. Designed with UC's human factors team and validated with NASA HERA analog astronauts in August 2025.

UI Architecture

Electron application connecting via Redis — no additional Docker containers or software required. Can be installed and run on any crew laptop without disrupting the GRAMS backend. Starts recovery analysis automatically in the background on failure detection. Input elements designed for use during high-g maneuvers with limited mobility.

HERA Test Scenario

Analog astronauts navigated a 50 km lunar surface route in a rover under time pressure, managing three failure modes: spacecraft exhaust plume, wheel slip, and meteorite impact. Two path options at 1 m/s, one nominal path at 2 m/s, with 30-minute switching penalties. Sensor updates every 10 seconds.

Key Testing Findings
  • In nearly all cases, the first user action was to initiate the autorecover process — prompting a design change to launch autorecover automatically on failure detection.
  • Participants showed high trust in AI action recommendations despite incomplete understanding of the underlying models.
  • Saliency plots were found intuitive and useful for identifying which sensors drove a detection decision.
  • Summary statistics (e.g., mean risk per action path) were preferred over detailed plots, which were still used to inspect sensor time series.
  • Panel navigation was the primary friction point; revised design groups all recovery steps under their parent failure event.
